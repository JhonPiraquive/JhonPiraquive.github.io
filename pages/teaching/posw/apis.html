¿Qué es una API?

Una API (Application Programming Interface) es un conjunto de definiciones y protocolos que permite que dos aplicaciones se comuniquen entre sí. Define qué operaciones están disponibles, cómo solicitarlas y qué esperar como respuesta, sin exponer la implementación interna.

Analogía: El menú de un restaurante es una API. Tú (el cliente) no entras a la cocina (el servidor/backend). Le das la orden al mesero (la API) usando el vocabulario del menú (los endpoints), y la cocina te devuelve lo que pediste (la respuesta). No sabes, ni necesitas saber, cómo se preparó.
Abstracción

Oculta la implementación interna.

Contrato

Define qué se puede pedir y qué se recibirá.

Interoperabilidad

Permite que sistemas distintos colaboren.

[App Cliente] │ │ HTTP Request │ GET /api/clima?ciudad=Medellin ▼ [API Gateway / Servidor] │ │ Procesa, consulta datos ▼ [Fuente de datos / Servicio] │ │ HTTP Response (JSON) ▼ [App Cliente recibe datos]

Tipos de APIs

Por accesibilidad
API Pública (Open API)

Accesible para cualquier desarrollador, generalmente con API key. Ejemplos: OpenWeatherMap, Google Maps, Stripe.

API Privada (Interna)

Solo accesible dentro de la organización. Conecta microservicios o sistemas internos. No expuesta al público.

API de Partner

Acceso restringido a socios comerciales mediante acuerdos. Ejemplo: APIs de pago entre plataformas.

Por arquitectura
REST

La más común. HTTP + JSON.

SOAP

XML estricto. Sistemas financieros.

GraphQL

El cliente define la respuesta.

gRPC

Alto rendimiento. Protocol Buffers.

Ver tipos-servicios-web.html para detalle completo de cada arquitectura.

Herramientas para trabajar con APIs

Postman
GUI

Plataforma completa para diseñar, testear y documentar APIs. Permite colecciones, entornos, mocks y tests automatizados.

Insomnia
GUI

Alternativa open-source a Postman. Soporta REST, GraphQL y gRPC. Interfaz limpia y rápida.

curl
CLI

Herramienta de línea de comandos para transferir datos con URLs. Disponible en todos los sistemas operativos.

HTTPie
CLI

CLI más legible que curl. Sintaxis simplificada, resaltado de sintaxis en terminal.

Swagger UI / OpenAPI
GUI (web)

Interfaz interactiva generada desde la especificación OpenAPI 3.0 del API.

Thunder Client
VS Code

Extensión de VS Code para testear APIs sin salir del editor.

Hoppscotch
GUI (web)

Cliente web de APIs open-source (hoppscotch.io). No requiere instalación.

REST Assured
Librería

Librería Java para testing automatizado de APIs REST en proyectos JUnit/TestNG.

Comandos curl comunes
# GET básico
curl https://api.ejemplo.com/productos

# GET con header de autenticación
curl -H "Authorization: Bearer TOKEN" \
     https://api.ejemplo.com/usuarios/me

# POST con body JSON
curl -X POST https://api.ejemplo.com/productos \
     -H "Content-Type: application/json" \
     -d '{"nombre": "Teclado", "precio": 150000}'

# PATCH
curl -X PATCH https://api.ejemplo.com/productos/42 \
     -H "Content-Type: application/json" \
     -d '{"precio": 130000}'

# DELETE
curl -X DELETE https://api.ejemplo.com/productos/42

# Ver headers de respuesta
curl -I https://api.ejemplo.com/productos

Buenas Prácticas de Diseño de APIs

Buenas prácticas
Usar sustantivos en los URIs, no verbos: /productos no /obtenerProductos.
Versionar la API desde el inicio: /api/v1/, /api/v2/.
Usar HTTPS siempre en producción.
Devolver códigos de estado HTTP correctos (201, 404, 422, 500...).
Incluir mensajes de error descriptivos en el cuerpo JSON.
Implementar paginación en colecciones: ?page=1&limit=20.
Documentar con OpenAPI/Swagger desde el día uno.
Implementar rate limiting y throttling.
Autenticar con JWT u OAuth 2.0, no con usuario/contraseña en cada request.
Anti-patrones
GET /eliminarProducto/42 (verbo en URI, método incorrecto).
Devolver siempre 200 OK aunque haya errores.
Exponer datos internos sensibles (contraseñas hasheadas, IDs de infraestructura).
No versionar y romper clientes existentes con cambios.
