¿Qué es el Modelo Cliente-Servidor?

El modelo cliente-servidor es un paradigma de arquitectura de red donde los roles están claramente separados: el cliente solicita servicios o recursos, y el servidor los provee. Es la base sobre la que opera la World Wide Web, el correo electrónico, las bases de datos en red y la mayoría de aplicaciones modernas.

Analogía: El modelo cliente-servidor es como un cliente en un banco. Tú (cliente) te acercas a la ventanilla (servidor) y haces una solicitud (request): "quiero retirar $200.000". El cajero (servidor) verifica tu identidad, consulta la base de datos, y te da la respuesta (response): los billetes o un error.

¿Cómo Funciona?

Flujo paso a paso al abrir https://ejemplo.com/productos:

1

El usuario escribe la URL en el navegador.

2

El navegador consulta un servidor DNS para resolver ejemplo.com a una IP: 190.25.80.42.

3

El navegador establece una conexión TCP con el servidor en el puerto 443 (HTTPS).

4

Se realiza el handshake TLS para cifrar la conexión.

5

El navegador envía la petición HTTP: GET /productos HTTP/1.1.

6

El servidor recibe la petición, la procesa (consulta DB, aplica lógica de negocio).

7

El servidor envía la respuesta: HTTP/1.1 200 OK con el cuerpo HTML o JSON.

8

El navegador recibe la respuesta y renderiza la página.

Resolución DNS

El navegador pregunta al resolver DNS por la IP de ejemplo.com y recibe una dirección IP.

TCP · handshake

Entre navegador y servidor: intercambio SYN / SYN-ACK / ACK para abrir el canal fiable (p. ej. puerto 443).

TLS · handshake

Negociación del cifrado (HTTPS) antes de enviar datos de aplicación.

HTTP request

Por ejemplo GET /productos HTTP/1.1 hacia el servidor de aplicación.

Consulta a la base de datos

El servidor ejecuta la lógica y consulta la BD (p. ej. SELECT …); recibe filas y construye la respuesta.

HTTP response

HTTP/1.1 200 OK con cuerpo JSON (o HTML) de vuelta al navegador.

Arquitecturas Derivadas

2 Capas (2-Tier)

Cliente
Servidor BD integrada en el mismo host

Ejemplo: aplicación de escritorio con acceso directo a MySQL.

Ventaja: sencillo. Desventaja: el cliente tiene acceso directo a la BD (inseguro), difícil de escalar.

3 Capas (3-Tier) — la más común en la web

1
Presentación Navegador / app móvil
2
Lógica (API) Servidor backend — Node.js, Spring, etc.
3
Datos PostgreSQL, MongoDB, Redis…

Ventaja: separación de responsabilidades, escalabilidad por capa, seguridad (la BD nunca está expuesta al cliente).

N Capas / Microservicios

Cliente
API Gateway
Servicio Usuarios ↔ BD Usuarios
Servicio Productos ↔ BD Productos
Servicio Pagos ↔ BD Pagos

Ventaja: cada servicio escala independientemente. Desventaja: mayor complejidad operacional.

Variantes del Modelo

Peer-to-Peer (P2P)

Cada nodo actúa como cliente y servidor simultáneamente. Ejemplos: BitTorrent, blockchain. No hay un servidor central.

Híbrido (Cliente-Servidor + P2P)

Un servidor central coordina, pero los clientes comparten datos entre sí. Ejemplo: Skype, Spotify (metadatos en servidor, streaming P2P).

Serverless

El código se ejecuta en funciones efímeras en la nube (AWS Lambda, Cloudflare Workers). El desarrollador no gestiona servidores; la nube escala automáticamente.
