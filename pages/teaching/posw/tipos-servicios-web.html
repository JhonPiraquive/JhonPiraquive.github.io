Tipos de Servicios Web

SOAP, REST, GraphQL, gRPC y WebSockets: cuándo usar cada arquitectura.

Explorar

SOAP — Simple Object Access Protocol

SOAP es un protocolo de mensajería basado en XML desarrollado por Microsoft en 1998 y estandarizado por la W3C. Define un formato de mensaje estricto con un sobre (envelope), cabecera y cuerpo. Usa WSDL (Web Services Description Language) para describir el contrato del servicio.

Características clave
Protocolo (no estilo arquitectónico): define reglas estrictas de formato.
Transporte: principalmente HTTP, pero también SMTP, TCP.
Formato: siempre XML. Sin excepciones.
Contrato explícito: el WSDL define operaciones, parámetros y tipos.
WS-Security: estándar de seguridad robusto (firmado y cifrado a nivel de mensaje).
Verbosidad alta: mensajes XML muy extensos.
Predominante en servicios financieros, bancarios y sistemas gubernamentales legacy.
Ejemplo de mensaje SOAP
<soapenv:Envelope
  xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
  xmlns:usr="http://api.ejemplo.com/usuarios">
  <soapenv:Header/>
  <soapenv:Body>
    <usr:ObtenerUsuario>
      <usr:id>42</usr:id>
    </usr:ObtenerUsuario>
  </soapenv:Body>
</soapenv:Envelope>

Casos de uso reales: integración bancaria interbancaria (ACH), servicios de la DIAN, SAP, sistemas de salud HL7.

REST — Representational State Transfer

REST no es un protocolo sino un estilo arquitectónico definido por Roy Fielding en su tesis doctoral del año 2000. Aprovecha HTTP tal como fue diseñado: URIs como identificadores de recursos, métodos HTTP como verbos de operación, y códigos de estado como resultado.

Seis principios
Stateless
  • Cada petición del cliente al servidor debe contener toda la información necesaria.
  • El servidor NO guarda información de peticiones anteriores.
  • Cada request es independiente.
Client-Server

Se separan responsabilidades:

  • Cliente: interfaz (frontend, app móvil)
  • Servidor: lógica y datos
Cacheable

Las respuestas del servidor pueden ser cacheadas para mejorar rendimiento.

  • Reduce carga en el servidor
  • Acelera respuestas
Layered System

La arquitectura puede tener múltiples capas intermedias:

  • Load balancer
  • Proxy
  • Gateway
  • Microservicios

El cliente no sabe si habla directo con el servidor o no.

Ejemplo:
Cliente → API Gateway → Microservicio → Base de datos

Uniform Interface

Todas las interacciones siguen reglas estándar.
Uso de HTTP methods:

  • GET → obtener
  • POST → crear
  • PUT → actualizar
  • DELETE → eliminar
  • Uso de URLs claras
Code on Demand (opcional)

El servidor puede enviar código ejecutable al cliente.

Generalmente JavaScript

Ejemplo:
El servidor envía un script JS que el navegador ejecuta.

Endpoints REST
GET    /api/v1/productos        → Lista productos
GET    /api/v1/productos/42     → Obtener producto 42
POST   /api/v1/productos        → Crear producto
PUT    /api/v1/productos/42     → Reemplazar producto 42
PATCH  /api/v1/productos/42     → Actualizar parcialmente
DELETE /api/v1/productos/42     → Eliminar producto 42
Respuesta JSON con HATEOAS
{
  "id": 42,
  "nombre": "Laptop Pro 15",
  "precio": 4500000,
  "stock": 12,
  "_links": {
    "self": "/api/v1/productos/42",
    "categoria": "/api/v1/categorias/3"
  }
}

GraphQL

GraphQL es un lenguaje de consulta para APIs y un runtime de ejecución, creado por Facebook en 2012 y open-sourced en 2015. A diferencia de REST, el cliente especifica exactamente qué datos necesita. Hay un único endpoint (generalmente POST /graphql) y la estructura de la respuesta espeja la estructura de la consulta.

Over-fetching (REST)

REST devuelve campos que no necesitas

El cliente define exactamente los campos.

Under-fetching (REST)

REST necesita múltiples requests

Una sola query anidada obtiene todo.

Versionado complicado

Romper clientes con cambios de API

El schema evoluciona sin versiones.

Query GraphQL
query ObtenerPedido($id: ID!) {
  pedido(id: $id) {
    id
    total
    cliente {
      nombre
      email
    }
    items {
      producto {
        nombre
        precio
      }
      cantidad
    }
  }
}
Mutation GraphQL
mutation CrearProducto($input: ProductoInput!) {
  crearProducto(input: $input) {
    id
    nombre
    precio
  }
}

gRPC — Google Remote Procedure Call

gRPC es un framework de RPC de alto rendimiento desarrollado por Google en 2015. Usa HTTP/2 para transporte y Protocol Buffers (protobuf) como formato de serialización binaria, lo que lo hace significativamente más eficiente que JSON/HTTP en términos de velocidad y tamaño de mensaje.

Serialización binaria

3–10x más compacto que JSON.

HTTP/2

Multiplexación, streaming bidireccional.

Contrato estricto

El archivo .proto define la interfaz.

Archivo .proto
syntax = "proto3";

service ProductoService {
  rpc ObtenerProducto (ProductoRequest) returns (ProductoResponse);
  rpc ListarProductos (ListaRequest) returns (stream ProductoResponse);
}

message ProductoRequest {
  int32 id = 1;
}

message ProductoResponse {
  int32 id = 1;
  string nombre = 2;
  double precio = 3;
  int32 stock = 4;
}

WebSockets

WebSockets es un protocolo de comunicación bidireccional y full-duplex sobre una única conexión TCP persistente. A diferencia de HTTP, una vez establecida la conexión mediante el handshake de upgrade, tanto cliente como servidor pueden enviar mensajes en cualquier momento sin necesidad de nuevas peticiones.

CLIENTE SERVIDOR |── GET /ws (Upgrade: websocket) ──►| |◄── 101 Switching Protocols ───────| | | | CONEXIÓN PERSISTENTE ABIERTA | | | |── {"tipo":"mensaje","texto":"Hola"}►| |◄── {"tipo":"respuesta","texto":"!"} | |── {"tipo":"ping"} ───────────────►| |◄── {"tipo":"pong"} ───────────────| | | |── close ──────────────────────────►|

Casos de uso: chat en tiempo real, notificaciones push, dashboards live, juegos multijugador, trading en tiempo real.

Ejemplo cliente JavaScript
const ws = new WebSocket('wss://api.ejemplo.com/chat');

ws.onopen = () => {
  ws.send(JSON.stringify({ tipo: 'unirse', sala: 'general' }));
};

ws.onmessage = (evento) => {
  const datos = JSON.parse(evento.data);
  console.log('Mensaje recibido:', datos);
};

ws.onerror = (error) => console.error('WebSocket error:', error);
ws.onclose = () => console.log('Conexión cerrada');

Rendimiento Bajo Medio Medio Alto Alto

Curva aprendizaje Alta Baja Media Media Baja
Ideal para Legacy/bancario APIs públicas web BFF, frontend flexible Microservicios internos Tiempo real
Regla de selección: ¿API pública para mobile/web? → REST. ¿Frontend necesita datos muy específicos? → GraphQL. ¿Microservicios internos de alto rendimiento? → gRPC. ¿Comunicación en tiempo real? → WebSockets. ¿Integración con sistema bancario legacy? → SOAP.
